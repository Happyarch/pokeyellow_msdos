## Context

The bag (ITEM) screen and its toss sub-menus are functional but mis-positioned, and
fixing them properly forces a decision about our overall windowing strategy. Two root
problems:

1. **Wrong horizontal projection.** The bag uses `IO_WX=87` (GB screen *centered* in
   our 320px viewport, offset 80px), but the already-corrected START menu uses
   `IO_WX=247` (GB screen *flush to the right edge*, offset 160px). The bag must use
   the same projection the START menu already does.
2. **One-box-at-a-time limit.** Our port renders the overworld BG at native 320×200
   from `wSurroundingTiles` (not a GB tilemap), so menus were put on the GB *window*
   layer (`render_window`, a single opaque rectangle). The original instead draws all
   menu boxes into one shared tile grid (`wTileMap`) and lets draw order resolve
   overlap — which is why it can show a list, a sub-menu, and a bottom dialog at once.
   We can't, today.

**Decision (chosen): unify the windowing model.** Rather than bolt "extra windows"
onto the single-window path, make an **ordered list of window descriptors** the *one*
way the window layer is drawn. Runtime cost is a wash vs. the additive approach (the
blit work is identical), so we pick on long-term cleanliness: the battle UI will need
a HUD + bottom dialog + a move/menu box simultaneously, and a unified descriptor model
builds that machinery once. The migration is kept low-risk by (a) a one-call shim for
the common single-window case and (b) a leak-proof "each screen rebuilds its own list"
rule.

## Coordinate projection — per-subsystem conventions

Translation is decided **per subsystem**, not globally, because different screens map
GB space into our viewport differently. Maintain a small convention registry; each
subsystem declares its screen transform and any forced spacing overrides:

- **Overworld UI (start / bag / party menus): TRANSFORM (confirmed).** Extended
  viewport (40×25) with per-element anchoring (rule below); mostly pure projection,
  with occasional spacing overrides allowed. The START menu (top-right) already
  established this.
- **Battle: GB-centered (TBD).** Keep the GB viewport default → center the whole
  160×144 GB screen in ours (X +10 tiles), with Y centered (+3.5 tiles), Y-scaled
  (144→200), or hand-tuned spacing — decided when battle is built. No per-element
  anchoring; one uniform transform.
- Future subsystems: add an entry when introduced.

### Tracking applied coordinate ops (translation / scale / override)
Record every non-identity GB→port coordinate op so divergences from faithful GB
coordinates stay auditable (like the existing `; BUG` / `; GLITCH` / `; TODO-HW` tags):
- **Inline tag at each placement site** — greppable, self-documenting. Format:
  `; PROJ <subsystem>: GB(<col,row>) <WxH> --(<op>)--> wx=.. wy=.. clip=..`
  e.g. `; PROJ overworld-ui: GB(4,2) 16x11 --(anchor=top-right, X+20, Y+0)--> wx=199 wy=16 clip=128`.
  Scale/override ops are tagged the same way (`--(Y-scale 144->200)-->`, `--(spacing override)-->`).
- **Central registry doc** — `docs/ui_projection.md`: the per-subsystem conventions
  above + an index of applied tags (can be regenerated by grepping `PROJ`). One place
  to see every transform/scale/override and which subsystem owns it.

### Anchor rule (for TRANSFORM subsystems, per axis, by developer intent)

The GB screen (20×18 tiles) sits inside our viewport (40×25), leaving **20 tiles
horizontal slack, 7 tiles vertical slack**. Per axis:

```
our_col = gb_col + H_SHIFT     H_SHIFT = 0 (left) | 10 (center) | 20 (right)
our_row = gb_row + V_SHIFT     V_SHIFT = 0 (top)  |  ~3-4 (center) | 7 (bottom)
wx = our_col_left*8 + 7 ; wy = our_row_top*8 ; clip_w = width*8 ; max_y = (our_row_top+H)*8
```

Examples: a **top-left** element translates neither (0,0); a **top-right** element
translates X only (+20, flush-right) and keeps Y (top edge is shared). The START menu
is top-right, so `our_col = gb_col + 20` reproduces its known `wx=247` — Python-verified.

**Process rule (per UI element):** infer the anchor per axis from pret intent, state
the inference, and confirm with the developer whether **X, Y, both, or neither** is
translated before placing it. Record the chosen anchor next to each element.

### Bag elements — anchor: TOP-RIGHT (X +20, Y +0) — CONFIRMED
Overworld UI = transform; the bag inherits the START menu's top-right frame. All bag
boxes sit flush against the GB right edge, measured from the top, so X translates +20
and Y stays. Under that anchor the original GB coordinates project to:

| Box (pret)          | top-left (col,row) | W×H total | our px x[..] y[..]   | wx  | wy |
|---------------------|:------------------:|:---------:|:-------------------:|:---:|:--:|
| Item list           | (4, 2)             | 16×11     | x[192,320] y[16,104]| 199 | 16 |
| USE/TOSS submenu    | (13, 10)           | 7×5       | x[264,320] y[80,120]| 271 | 80 |
| Toss YES/NO         | (14, 7)            | 6×5       | x[272,320] y[56,96] | 279 | 56 |
| Quantity (unpriced) | (15, 9)            | 5×3       | x[280,320] y[72,96] | 287 | 72 |

`TextBoxBorder` takes interior W×H (total = interior + 2). Item names at (6,4)/rows
6,8,10; list cursor (5,4); USE/TOSS items (15,11); YES/NO items (15,8).

**Reference frame (verified):** the original places every box in ABSOLUTE screen
coordinates — `data/text_boxes.asm` stores absolute corners (`LIST_MENU_BOX 4,2,19,12`,
`USE_TOSS 13,10,19,14`, …) and `hlcoord x,y` = `wTileMap + y*20 + x`. There is no
nested/parent-relative frame; only the text/cursor *inside* a box are box-relative
offsets. So each box is translated independently by its own anchor — but since all bag
boxes share the top-right anchor, they all translate identically (+20 X, +0 Y) and stay
coherent. (No relative-coordinate machinery needed.)

## Approach (detail)

### Phase 1 — Unified window descriptor list (the one window path)
- Define a **window descriptor**: `{ wx, wy, clip_w, max_y, tilemap_base, src_start_row }`.
  Add a global ordered array `g_windows[MAX_WINDOWS]` (MAX ≈ 6) + `g_window_count`.
- `render_window` (`dos_port/src/ppu/ppu.asm`) takes one descriptor (it already
  decodes a tile rect and branches `GB_TILEMAP0/1` at ppu.asm:660/663, and resets its
  line counter per call). Change it to read `wx/wy/clip_w/max_y/tilemap_base/start_row`
  from the descriptor instead of the fixed globals.
- `frame.asm` present step (`frame.asm:76`, the **only** caller): loop
  `for i in 0..g_window_count-1: render_window(&g_windows[i])`. `count==0` ⇒ nothing
  drawn. Later entries draw on top (painter's order). No transparency/z-buffer needed —
  each box is a solid bordered rect, so a descriptor tightly sized to its box exposes
  no blank area; overlap resolves by order.
- **Leak-proof discipline:** a screen sets the *entire* list when it takes control and
  whenever its window state changes; it never relies on prior contents. This replaces
  any "remember to clear on exit" footgun — the next owner always fully redefines the
  list. Provide a `set_single_window(wx,wy,clip_w,max_y,tilemap,start_row)` helper that
  sets `count=1` and fills entry 0 (the migration target for current single-box code).

### Phase 1b — Migrate existing consumers to the list (mechanical)
Each currently writes the window globals then presents; replace those writes with a
`set_single_window(...)` call (same numbers), or `count=0` to hide. Files (from grep):
- `dos_port/src/text/text.asm` (PrintText dialog) and
  `dos_port/src/engine/overworld/map_sprites.asm` (NPC dialog) → one bottom-box descriptor.
- `dos_port/src/engine/menus/start_menu.asm`, `…/party_menu.asm` → one descriptor each.
- `dos_port/src/engine/overworld/overworld.asm` (the "hide window" paths) → `count=0`.
- `dos_port/src/init/init.asm` → initialize `count=0`.
Behavior must be byte-for-byte identical for these (verified via their existing render
output); only the bag uses `count>1`.

## Impact on the existing windowing strategy
- **Single call site.** `render_window` is invoked only from `frame.asm:76`, so the
  pipeline change is centralized.
- **Semantic shift:** the renderer stops reading the loose globals (`IO_WX`/`H_WY`/
  `g_win_clip_w`/`g_win_max_y`) directly and reads descriptors instead. The GB regs
  `IO_WX`(rWX)/`H_WY`(rWY) may remain for faithfulness but are no longer the render
  source of truth; `set_single_window` is the bridge during migration.
- **New shared rule:** screens fully (re)define `g_windows`/`g_window_count` on
  entry/state-change. Document it where the array is defined.
- **Tilemap allocation:** descriptors name their tilemap explicitly; keep menus on
  `GB_TILEMAP1` as today and use `GB_TILEMAP0`/other rows for the bag's extra boxes,
  avoiding the dialog's `GB_TILEMAP1` rows 0–5.
- **Future battle UI** reuses the same list (HUD + dialog + menu) — build only what the
  bag needs now; don't pre-build battle-specific descriptors.
