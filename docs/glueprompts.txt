HOW TO USE
- Each WAVE is split into THIRDS (a / b / c) so you can feed one third per fresh
  session. A non-save cross-cut is appended to each wave's tail (the end of its
  "c" third) as a change-of-pace; the SAVE SYSTEM is last, after every wave.
- These are TEMPLATES. The handoff agent that finishes a third should edit the
  next third's prompt in place — prune drift, fix file paths / pret refs, paste
  "resume here" notes — before you feed it.
- Every prompt assumes the agent reads CLAUDE.md at session start (it does).

STANDING CONVENTIONS (true for every prompt; CLAUDE.md is authoritative)
- Faithful pret translation; keep pret label names; EBP-relative GB memory model +
  register map per CLAUDE.md. Emit "; TODO-HW:" at any $FF__ I/O boundary.
- DATA vs CODE (two-tier rule): generators write ONLY data-only assets/*.inc;
  never hand-edit a generated .inc; per-entry behavior lives in hand-authored
  .asm; tiers link by id/index; data deviations go in the generator or a
  BUG_FIX_LEVEL code block.
- Tag known bugs "; BUG(level):" (docs/bugs_and_glitches.md) and exploitable
  glitches "; GLITCH:"; use BUG_FIX_LEVEL %if blocks.
- WRAM/constant addresses are sym-pinned — reuse gb_memmap.inc / gb_constants.inc
  aliases; add new ones only with a pinned source (note it).
- Gen-2 forward-compat: NEVER shrink/realign/repurpose party (44B) / box (33B)
  struct bytes; carry offset-7 (catch-rate / GS held item) verbatim.
- VALIDATION: EXE needs a DPMI host (local Arch has DOSBox-X + CWSDPMI). Backend
  logic → native ELF32 harness (nasm -f elf32 + tiny _start: set EBP to a buffer,
  stub data, call routine, sys_write result). Always: nasm -f coff per file, then
  `make check`, then full `make SKIP_TITLE=1` link. Trap: `make clean` leaves
  src/{debug,engine/debug}/*.o — rm them before switching DEBUG_* flags.
- BOOKKEEPING per task: plan stage [x], docs/translation_log.md entry, update the
  plan Handoff section. Plan fully done → archive (git mv current_plan_<t>.md
  docs/plans/<t>.md) + fix CLAUDE.md "Currently active plans".
- GIT: commit only when asked; branch off master first; Co-Authored-By trailer;
  assets/ is gitignored but force-tracked → `git add -f` new generated .inc.

CROSS-CUT PLACEMENT (note): there are only three non-save cross-cuts
(party_popup Stage 5, audio HAL, GBC palette), so one is appended per wave tail
(not two); the save system is the final section. Rebalance if you prefer.

------------------------------------------------------------------------------
PARALLELIZATION (sonnet subagents) — don't step on shared files
------------------------------------------------------------------------------
The parallelizable, expensive part is authoring + native-validating each
DEDICATED .asm. The contention is a small set of SHARED files every task wants to
touch. RULE: a parallel subagent edits ONLY its own dedicated .asm (+ its scratch
harness). It must NOT edit any shared file — it REPORTS the deltas in its final
message, and the orchestrator applies them serially.

ORCHESTRATOR-OWNED (shared) files — never edited by parallel subagents:
  - dos_port/Makefile                       (BATTLE_SRCS/POKEMON_SRCS lines, rules)
  - dos_port/include/gb_memmap.inc, gb_constants.inc   (WRAM/constant aliases)
  - dos_port/src/data/battle_data.asm       (shared data globals)
  - docs/translation_log.md                 (append-only log)
  - docs/current_plan_*.md                  (stage checkboxes / handoff)

PROTOCOL:
  1. PREP (orchestrator, serial, before dispatch): add every WRAM/constant alias
     the wave's tasks will need to gb_memmap/gb_constants (predictable from the
     pret refs) and commit, so subagents assemble against a stable include set.
  2. PARALLEL (subagents): each writes its dedicated .asm + a scratch ELF32
     harness and validates with `nasm -f coff -I include/ -I .` + the harness
     (NOT `make`, which needs the shared Makefile). Final message REPORTS:
     (a) the exact Makefile source-list line to add, (b) any extra alias it needed
     (pinned address), (c) its translation_log entry, (d) the plan checkbox.
  3. INTEGRATE (orchestrator, serial): apply each agent's reported deltas, run
     `make check` + full link, commit per task. (Or give each agent
     Agent isolation:"worktree" and merge out ONLY its new .asm — do not merge its
     shared-file edits.)

PER-WAVE:
  - WAVE 1 — FULLY PARALLEL. The six tasks write DISJOINT .asm:
      1 bills_pc.asm | 2 (new) residual_damage.asm | 3 (new) trainer_ai.asm +
      read_trainer_party.asm | 4 experience.asm | 5 (new) evolution.asm |
      6 (new) effects.asm (JumpMoveEffect).
    Integration-order edges (NOT file contention): 5 integrates after 4 (level-up
    flow); 6 references move_effects/*.asm read-only (safe parallel); 3 may add a
    trainer-data global to battle_data.asm → report it. Launch 1,2,3,4,6 together;
    5 after 4 (or author 5 in parallel, integrate last). party_popup cross-cut is
    UI → solo/serial.
  - WAVE 2 — NOT parallelizable. The battle front-end is UI + manual-verification;
    all stages share the battle core/render files + battle WRAM. Run 2a→2b→2c
    sequentially, one agent, user checks each FRAME.BIN. EXCEPTION: the audio HAL
    cross-cut is disjoint (src/audio/* + audio_hal.inc) → a separate agent can run
    it alongside the battle work once that subsystem starts.
  - WAVE 3 — PARTLY PARALLEL. 11 item USE (item_effects.asm + ItemUsePtrTable) is
    independent of the script subsystem → run it parallel to the script thread.
    12/13/14 share the script files (pallet_town.asm, run_map_script, map scripts)
    → keep that thread SERIAL (12→13→14). GBC-palette cross-cut (boot/video.asm)
    is independent → separate agent anytime.
