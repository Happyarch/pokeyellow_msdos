# Current Plan: Items Layer

The item/bag system, sequenced after the Pokémon data engine
(`current_plan_pokemon_engine.md`) and before battle. Strategy per the user
directive: **do everything that doesn't touch the GUI; stub the UI parts and note
them.** The bag/PC menus, item-toss prompts, and "used ITEM!" text are UI; the
inventory bookkeeping and most item *effects* are not.

## Conventions
- Same as the pokemon plan: faithful pret translation, sym-pinned WRAM addresses
  (`origin/symbols:pokeyellow.sym`), native ELF32 + `gcc -m32` validation, track
  via `dos_port/tools/work_queue`.
- Static item data (names, prices, effect tables) → generated by Python tools.

## Stages

- [x] **Stage 1 — bag/PC inventory bookkeeping (no UI).**
  `src/engine/items/inventory.asm`: `AddItemToInventory_` /
  `RemoveItemFromInventory_` (count + (id,qty)-pair + $FF-terminator arrays;
  stacking, 99-cap overflow to a new slot, slot removal + shift-up, bag-vs-PC
  capacity). Replaced the swarm draft (it tested the wrong byte for an empty
  inventory — advanced hl before the zero-count check). Added WRAM aliases
  (`wNumBagItems`/`wBagItems`/`wNumBoxItems`/`wBoxItems`/`wCurItem`/`wItemQuantity`/
  `wMaxItemQuantity` + the menu-state bytes it resets) and `BAG/PC_ITEM_CAPACITY`.
  Native-validated (add/stack/overflow/full/remove-partial/remove+shift). Wired
  (`ITEMS_SRCS`).

- [ ] **Stage 2 — item static data.** Generate item names + prices
  (`data/items/*`), and the `ItemUsePtrTable` mapping. Names/prices are pure data.

- [ ] **Stage 3 — non-UI item effects.** The data-only parts of `ItemUse*`:
  e.g. potion HP math, status-cure flags, stat-boost (X Attack) application, PP
  restore, rare-candy level-up (calls the pokemon engine). Stub the surrounding
  text/animation/menu UI; keep the WRAM mutations. Many of these call routines
  the pokemon engine already provides (CalcStats, WriteMonMoves, …).

- [ ] **Stage 4 — bag operations glue.** Toss/use/give dispatch logic minus the
  menus; vending/mart buy/sell math (`subtract_paid_money` exists as a draft).

## Stubs / deferred (UI)
- Bag & PC item menus, item list rendering, "TOSS how many?" / "throw away ITEM?"
  prompts, "used ITEM!" / "it won't have any effect" text, item-use animations.
  These consume the menu-state bytes the non-UI routines already maintain.
